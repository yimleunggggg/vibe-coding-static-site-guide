# 目标与阶段

## 要实现什么

| 能力 | 自动化效果 |
|------|------------|
| **日报** | 每天拉 GA4（昨日+7日）+ GSC（28天，滞后2–3天）+ 静态自检 → `reports/daily/` |
| **半月报** | 每月 1/15 日拉 28 天汇总 → `reports/*-reminder.md` + Issue |
| **周报方案** | 每周一生成 `proposals/YYYY-WW-proposal.md` + Issue，**你 approve 后才改站** |
| **通知** | 可选 ntfy + Resend 邮件；默认 GitHub Issue Watch |
| **不改站** | 日报/半月只读数；执行优化需你确认 |

屋久岛场景：工具站（时刻表/上岛交通），不堆旅游攻略；新站 GSC 展示 0 属正常节奏。

## 分几个阶段

```text
阶段 A（人工一次性）  GSC 验证 + GA4 埋点 + sitemap
        ↓
阶段 B（授权）        Cloud API + 服务账号 + OAuth + GitHub Secrets
        ↓
阶段 C（通知，可选）  Resend + DNSPod + ntfy
        ↓
阶段 D（云端跑）      Actions：daily / biweekly / weekly
        ↓
阶段 E（你决策）      Issue approve → PR（待接 workflow）或 Cursor
```

## 不必依赖

- **Cursor**：仅复杂改版时用；日常 approve 用手机 GitHub 即可
- **本机开机**：Actions 在 GitHub 云端 cron 执行
